## What was tested

Retrain v3 (LightGBM, 47 features, same hyperparameters) on a NEW label
matching the live production target:

```
label_hit_1R_10d = 1 IFF
    high reaches close + 2*ATR20  BEFORE
    low touches  close - 2*ATR20  WITHIN
    next 10 trading days
```

This label aligns with the live IdeaBridge config (1R primary target,
2*ATR stops, 10-day max hold) — what the system actually trades.

## Interpretation

1. **The 10-day path-dependent target is too hard for v2 features.**
   Our features were engineered for 3-5 day forward returns. Pushing
   the window to 10 days dilutes signal under noise. Adding the path
   constraint ("up before down") makes it harder still.

2. **The "Triple Lock edge" is the FILTER, not the model.** When we
   showed Triple Lock + 1R/2*ATR/10d had +0.024 R/trade net expectancy
   on n=1,063 (rr_analysis_live_config), that was a POPULATION
   statistic from the rule (conviction>70 + ml_v2>70 + F4 insiders>=1
   + accumulation phase), not skill from v3. v3 adds zero on top of
   the rule for this frame.

3. **The same model retrained on shorter labels still works.** The
   prior 0.626 AUC on 3pct_3d is real signal — just not at the
   horizon and stop frame we're trading.

## Decision

- **Do NOT calibrate this model.** Calibrating a 0.51 AUC produces a
  flat mapping to base rate. F2 (probability_calibration.py) exists
  and is tested but won't be applied to this artifact.
- **Do NOT deploy this model in the live pipeline.** Triple Lock as
  a rule-based filter is fine on its own.
- **Next experiment:** retrain on `label_hit_1R_5d` (same R-frame,
  5-day window). If AUC > 0.58 we have a usable 5-day model. If not,
  retire v3 for this production target and accept Triple Lock alone.
